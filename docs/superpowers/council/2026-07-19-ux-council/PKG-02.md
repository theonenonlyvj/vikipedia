## Changes
In live prod the `<dl className="run-metrics">` (RaceMode.tsx:93-108, Clicks/Timer/Target chips) never paints, while its sibling PathStrip — gated on the same `session` truthy check — does. Six lenses independently confirmed this against mockup-race-flow-v3's 'Timer + clicks always on' contract. Debug systematically: (a) is prod running this code at all (stale deploy)? (b) is the `.race-hud` grid's 'metrics' row (styles.css:1020-1050) collapsing to zero height or being clipped by the sticky/backdrop-filter container? (c) is the session prop threading from RaceFlow stale? Restore a compact always-visible HUD row ('0:14 · 3 clicks' style) at every breakpoint; consider dropping run-metrics' duplicate Target chip since PathStrip already renders one. While in this screen, fix the WCAG AA failure: `.article-heading span` renders #9fb8bd muted-label color on the white article card (~2.1:1) — scope a dark color (e.g. #54595d, matching .attribution) to the light article-panel context. Add an automated visual/e2e assertion that Timer and Clicks are visible and non-empty during an active run so this can never silently regress again.

- A test asserts .run-metrics has non-zero rendered height and non-empty Timer/Clicks text while a session is active
- The challenge label over the white article panel measures ≥4.5:1 contrast

Two concrete problems with the package as written, both worth fixing before handoff:

1. Diagnostic hypothesis (c) is logically inconsistent with the code and will waste the timebox. RaceMode.tsx:93 (`{session ? <dl className="run-metrics">...`) and :119 (`{session ? <PathStrip .../>`) gate on the *identical* `session` value in the same render pass — if `session` were stale/null, PathStrip could not render either, but the screenshots show PathStrip rendering fine while run-metrics doesn't. RaceFlow.tsx passes `session={session}` straight through with no transformation. "Is the session prop threading stale" should be dropped as a hypothesis; it cannot explain the observed divergence.

2. The acceptance criterion "A test asserts .run-metrics has non-zero rendered height" is not achievable with this repo's actual test infra. vitest.config.ts uses `environment: "jsdom"` (confirmed), and there is no Playwright/Cypress/real-browser test tool in package.json devDependencies, no existing `*.test.*` file for RaceMode, and no a11y tooling (no axe anywhere in the repo) despite the commit message for dd0b2f4 (today, same file, "stable race HUD first paint") claiming ad hoc "Playwright/WebKit" verification outside the committed suite. jsdom does not run a real layout engine — `getBoundingClientRect`/`offsetHeight` return 0 regardless of CSS — so a literal "non-zero rendered height" assertion will either be unimplementable or silently no-op-pass. Split this into (a) a jsdom-feasible test asserting the `.run-metrics` node is present with correct, non-empty Timer/Clicks text whenever `session` is truthy (catches "JSX conditional/markup regresses to not rendering at all") and (b) an explicit manual/visual verification step for the actual suspected CSS-rendering defect, or flag that a real acceptance test for "renders on screen" requires adding browser-based test infra (separate scoping decision, not silently folded into this M estimate).

Additional finding worth surfacing: today's commit dd0b2f4 ("fix(ui): ... stable race HUD first paint", same day, touches this exact `.race-hud`/`.run-metrics` grid) claims Playwright/WebKit-verified fix for HUD instability and explicitly says "only the metrics row's height appears/disappears underneath [End Run], which is an ordinary height change" — implying run-metrics *did* render in that verification. That directly contradicts the live screenshots. This makes "confirm whether the currently live Cloudflare Pages deploy actually includes commit dd0b2f4/381902a/d8da396 (all same-day)" the single highest-leverage diagnostic step — should be made explicit step zero, ahead of any code speculation, since a fresh `npm run build` off current HEAD produces clean, structurally correct CSS (`.race-hud .run-metrics{grid-area:metrics}`, no conflicting rules) with no obvious static bug.

Minor scope nit: RaceFlow.tsx is listed under "files" but the package's "changes" text describes no edit there — given the spec's explicit warning that App.tsx/race-flow files are "the single highest merge-conflict-risk piece of the whole project" requiring a coordinated merge window with Codex, the package should state RaceFlow.tsx is for diagnostic reading only (confirming prop passthrough), not an edit target, so an implementer doesn't touch it unnecessarily.

None of this changes the P0 priority or the core fix direction — the bug is real, well-evidenced, invariant-restoring, and the code-level diagnosis area (RaceMode.tsx + styles.css) is correct and appropriately narrow with zero risk to timing/lifecycle/routing/server code. This is an amend, not a drop.
