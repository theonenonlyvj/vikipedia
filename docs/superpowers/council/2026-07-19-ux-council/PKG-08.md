REASONING: Hard invariants: PKG-08 does not touch race timing, run lifecycle, ranking, identity-timing, path-hiding, or pre-drop honesty — it's a pure presentation/routing change. No invariant violation found there.

Spec-fidelity of the core premise is contested, though. The package claims "mockup-race-flow-v3 scopes the 'no tabs here — fully focused' treatment to step 2 (active race) only, keeping header + nav on steps 1 and 3." The written, council-ratified spec disagrees: the Race-flow section's own heading is "### Race flow (full-screen takeover, no global chrome)" — applied to the whole flow, not scoped to step 2 — and step 1 (Preview) is spec'd with its own dedicated escape hatches ("← Back" top-left, "Not feeling it? See other challenges ›"), which only make sense as bespoke controls if the global bottom nav is *not* present to serve that role. PreRacePreview.tsx's own doc comment (lines 4-8) ties those two buttons directly to invariant 3 as "free exits" — current code already matches the spec text, not a divergence from it. Before restoring chrome to Preview/Results, this conflict needs to be resolved against the actual mockup image and Vijay's intent (a mockup can show a scoped treatment the spec text hasn't caught up to, or the package can be misreading the mockup) — this is a design-authority question, not something to route through as-is.

Implementability gap: the `files` list (RaceFlow.tsx, styles.css) is incomplete for what's actually described. AppShell.tsx has a load-bearing doc comment (lines 37-39): "App.tsx keeps `<RaceFlow>` rendered above/outside this component entirely... so this file never needs to know about the race flow at all." Embedding Preview/Results inside AppShell breaks that documented invariant and requires real AppShell.tsx + App.tsx changes: `raceEngaged` (App.tsx:302-306) must split into a chrome-free flag (RaceMode + recovery only) vs. an embedded-race-content flag consumed by AppShell.

That split introduces an undefined interaction surface: `selectMode`/`onSelectMode` (App.tsx:684-692, wired into AppShell's nav buttons) does not clear `raceStage`, `race.phase`, or `dnfResult` — only `exitRaceFlow`/`exitCompletedRaceTo` (App.tsx:829-839) do that, and those are currently invoked only by Preview's own Back/See-other-challenges and Results' own Play-another/Show-board/Show-challenges buttons. If bottom-nav buttons become visible+tappable during Preview/Results without being rewired through those exit functions, a nav tap will desync `mode` from `raceStage`/`phase`, likely leaving the race content rendered while the nav shows a different tab "active" (or vice versa). None of `changes`/`acceptance`/`risk` mention this.

Separately, AppShell's first-visit teaching gate (`showTeachingGate`, AppShell.tsx:164-168) is computed purely from `visibleMode` with no race-flow awareness. Since `mode` persists unchanged through Preview/Results (openRacePreviewFor doesn't touch it), a first-time player opening the Daily hero from Home would have mode="home" during Preview — risking the rules-strip rendering on top of the Preview/Results content unless AppShell is explicitly made race-aware.

Test-requirement gap: App.test.tsx already has two tests that explicitly assert the opposite of the new target state — "(a) + (e) renders zero global chrome... once race.phase is active" and "keeps the takeover engaged and shows Results (not the shell) immediately on completion" (App.test.tsx:2732-2757) assert `.closest(".race-takeover")` and absent nav specifically for Results. These must be deliberately rewritten, not just incidentally broken, and new coverage is needed for: nav-tap-during-Preview/Results correctly exits and clears state; teaching gate suppressed while race content is embedded; RaceMode itself stays chrome-free (regression guard for the one piece that should NOT change).

Effort is under-rated at M given the above — this is an App.tsx state-machine split plus an AppShell.tsx architecture change that its own doc comments say shouldn't be necessary, not a component-local tweak.

The dark-reskin of `.pre-race-copy`/`.target-preview-blurb`/`.target-preview-attribution` (item 2) and the desktop back-button composition fix (item 3) are sound, low-risk, and spec-aligned regardless of how the chrome question resolves — dark identity is mandated app-wide ("Explicitly out of scope / visual style — RESOLVED... dark near-black ground... New modes apply this existing identity consistently") and the white Wikipedia-chrome preview card is a legitimate, evidence-backed defect independent of nav placement.
