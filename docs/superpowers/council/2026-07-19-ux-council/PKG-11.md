## Judge A (amend) — design integrity
AMENDMENTS: 1. Drop or narrow the Boards time-window tab-label claim ("mix spelled-out and abbreviated labels with no stated convention"). Verified in code (Boards.tsx:20-29: Today/Yesterday spelled out, 7d/30d abbreviated, Lifetime spelled out): this reads as a defensible, common analytics convention — single-day boards get words, rolling-window boards get compact "Nd" notation, Lifetime is unbounded so no number attaches. The package's own remedy doesn't even touch the tab labels it complains about; it only rewords the "Rolling {segment}" prose line beneath them. Rescope this bullet to just the prose-string fix ("Rolling 30d" → "Rolling 30 days") and drop the implication that the tabs themselves need rework — otherwise a fixer may go rename tabs that don't need it.

2. Downgrade the "clk" bullet's framing. It is not leaked jargon like "ranking guard" — it's spec-ratified shorthand: docs/superpowers/specs/2026-07-18-ux-redesign-modes-design.md explicitly mandates "0:38 · 5 clk" (invariant 1, lines 43-44, restated at 187-189, 197, 241, 333) for space-constrained rows (leaderboard rows, state chips, card meta lines). The package's actual fix text already scopes this correctly (introduce once, don't expand everywhere), so no execution risk — just re-tag this as P3 polish, not grouped with the real "no jargon" jargon (ranking guard) it currently sits next to in `covers`.

3. Split the You-tab item's effort out of "S." Confirmed in App.tsx (lines 562/566/571) and You.tsx (lines 49-91): `accountStatsProjection.stats` is `null` both while loading AND after a fetch error — there's no existing signal to distinguish "loading" from "errored" from "guest/no account," so "a real loading state instead of bare '-'" requires new state plumbing in App.tsx, not a copy edit. Likewise "consolidate the nine stacked blanks into the Boards-style progress framing" is a new empty-state layout decision (what does a zero-play account's Stats panel actually look like?), i.e., component-level work. Recommend either (a) re-tagging this specific sub-item as effort M and letting the rest of the sweep ship at S, or (b) descoping it in this package to just "show 0 for confirmed-zero totals, keep 'No data yet.' — matching StatsList's own convention — for the unresolved/guest case" (already achievable with the current null-vs-object distinction, no new plumbing needed) and punting true loading-vs-error UX to its own ticket.

4. Fold "footer must never collide with the bottom nav on any mode screen" into the acceptance list so the footer-placement decision also fixes this, not just the tone.
