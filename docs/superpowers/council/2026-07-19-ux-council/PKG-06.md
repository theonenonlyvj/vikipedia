## Changes
This is the direct answer to 'the landing/home page doesn't line up with our original design' (mockup-home-stateful-v2 is always populated: hero + streak/avg chips + yesterday's board). (1) Kill the hollow state: Home.tsx:241 suppresses the 'Yesterday's results' BoardSnippet exactly when the hero IS yesterday's daily (`!yesterdayIsHero`) — the current prod state. Render the hero's already-fetched `heroBoardMatches` (Home.tsx:158) as the recap snippet in that branch; if no prior daily exists at all, fall back to a compact Boards summary. Home must have a board visible in every pre-play state. (2) StreakTrendRow (Home.tsx:307-321) returns null for guests/zero stats; give it a progress-framed empty state ('Start your streak today') instead of nothing. (3) Teaching gate per mockup-onboarding-v2: replace the one-liner with the numbered strip ('1 Start on an article · 2 Tap links inside the page · 3 Reach the target fast') for never-played visitors, auto-hiding after their first finished race, and add the spec's 'No account needed to look around.' line. Phase 2 note: if the numbered strip needs design time, ship items 1, 2, and the reassurance line first — they are the bulk of the visual parity. (4) Sparse layout: give `.home-layout` viewport-height distribution (min-height: calc(100dvh - chrome) + content alignment) so the mobile CTA doesn't strand in the top third above a void.

## Acceptance criteria
- In the fallback state (hero = yesterday's daily, guest viewer), Home shows hero + streak row (or its empty state) + a populated yesterday's board — side-by-side comparable to mockup-home-stateful-v2's 'BEFORE' frame
- First-visit Home shows the 3-step strip and 'No account needed to look around.'; both disappear after one finished race
- 390x844 screenshot shows no viewport-dominating dead space below the last content block

## Risk
Hero board reuse must handle the zero-matches case gracefully; teaching-strip auto-hide needs a persisted 'has finished a race' flag that survives reloads for guests.

## Judge B (amend) — engineering risk
AMENDMENTS: 1. DROP the client-persisted "has finished a race" flag for guests. The package's own risk note calls for "a persisted 'has finished a race' flag that survives reloads for guests" to drive the teaching-strip auto-hide. This directly violates the ratified spec's migration note (iii): "the first-finish teaching gate must be derived from account stats (races completed > 0), not device-local storage" (docs/superpowers/specs/2026-07-18-ux-redesign-modes-design.md:281). It's also unnecessary: src/domain/teachingGate.ts's `shouldShowTeachingGate` already implements exactly this — server-derived, with an explicit `hasIdentifiedSession` guard to avoid the "flash on load/stuck on error" bug (its own doc comment calls this "the M1 fix"). Ship the footer-line copy addition against the existing gating logic unchanged; do not add any storage.

2. DROP or re-route the "numbered strip" teaching-gate redesign. The package proposes replacing "Two articles. Links only. Beat the clock. (how to play)" with a visible 3-step strip ("1 Start on an article · 2 Tap links inside the page · 3 Reach the target fast"), citing mockup-onboarding-v2. But the ratified spec text is explicit and different: '"Two articles. Links only. Beat the clock." (how to play) — the parenthetical opens a quick-dismiss popup with fuller rules...' (spec lines 98-103). src/components/TeachingGate.tsx already implements this one-liner + popup pattern near-verbatim. The numbered strip is mockup-only content the ratified .md appears to have deliberately simplified away (the spec explicitly documents other such cuts, e.g. "Cut from earlier drafts: the rivalry/rematch strip"). Implementing the mockup's version instead of the spec's would be undoing a ratified simplification without a product decision to do so. Keep the package's item (3) narrowed to just the missing footer line: 'No account needed to look around.' (spec line 103) — confirmed absent from TeachingGate.tsx today, and correctly the one clean gap here. If the team wants the fuller numbered strip, that's a separate product call, not an engineering default.

3. ADD explicit test requirements — none are specified today, and Home.tsx has no existing test file (confirmed via search). Require: (a) a test/case verifying the pre-drop recap renders `heroBoardMatches` when `yesterdayIsHero` is true, including the empty-board case (BoardSnippet already renders "No completed runs yet." at rows.length===0 — confirm the new branch exercises that path rather than reinventing an empty state); (b) a test for StreakTrendRow's new empty-state copy that explicitly distinguishes "stats loaded and truly zero" from "stats still null because not-yet-loaded or fetch errored" — this is the exact ambiguity teachingGate.ts/teachingGate.test.ts already had to solve once (their comment calls it "the M1 fix"); without an equivalent guard here, the new copy can flash for accounts whose stats simply haven't arrived yet.

None of the above touches race timing, run lifecycle, App.tsx routing latches, or server queries — confirmed the package's file list (Home.tsx, TeachingGate.tsx, styles.css) stays clear of RaceFlow/RaceMode, App.tsx's run state machine, and any server/query code, so blast radius stays low once amended.
