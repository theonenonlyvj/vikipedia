## Changes
(1) P0 — the overlap smear: at ≥880px the ranked trend row renders name and 'avg #X.X (N dailies)' superimposed ('theeoge#h0ly3jdailies'). Cause: `.board-snippet li` and its only child `.trend-row-toggle` are BOTH 3-column grids with the same template (styles.css:874-889, 1554-1560, desktop override 2579-2584), so the button auto-places into the li's first 2.4em track and its inner columns overflow it. Fix: give `.trend-row-toggle` `grid-column: 1 / -1` (or drop grid from the li when it wraps a toggle), give the name column `minmax(0,1fr)` with ellipsis — mirroring the codebase's own 'Bug A fix' at styles.css:911-933 that was applied to unranked rows but never ranked ones. Verify at 880/1024/1440. (2) Segment control: make all 5 segments fit at 375-390px via the narrow-width font/padding pattern the spec's council notes called for, or move Lifetime behind 30d (spec Open Question 1's fallback). If it still scrolls at very narrow widths, make the edge fade scroll-aware or add an inert end spacer so a selected 'Lifetime' pill never renders half-faded, and keep a passive 'more' hint. (3) ARIA: complete the tablist pattern (roving tabindex, ArrowLeft/Right, aria-controls + role=tabpanel) or drop tab roles for the aria-pressed toggle-group pattern .mode-nav already uses correctly. (4) Give `.trend-row-toggle` min-height 44px (currently min-height:0/padding:0). (5) Move 'Race today's daily' below the leaderboard/DNF block per mockup-boards-trends' see-the-board-then-commit order.

## Acceptance criteria
- Desktop 7d/30d ranked rows are fully legible (no overlapping glyphs) at 880, 1024, and 1440px; long names truncate with ellipsis
- At 390px all five segments are visible without scrolling (or Lifetime is one tap behind 30d); a selected end pill never renders faded
- Keyboard/AT behavior matches the announced role (arrow keys work if tabs, or no tab semantics); own trend row is ≥44px tall
- Boards Today renders the CTA after the board content

## Risk
Segment fitting must hold at 320px; re-verify the existing programmatic scrollIntoView deep-link behavior after layout changes.

REASONING: Verified the core diagnosis against code and screenshots — it holds up.

CONFIRMED CORRECT:
- Grid-in-grid overlap: `.board-snippet li` (styles.css:1554-1560, desktop override 2579-2584) and `.trend-row-toggle` (874-889) are both 3-col grids with the same template, and the toggle is the li's ONLY child. Default CSS grid auto-placement gives an unplaced item a span of 1, so the button gets squeezed into the li's first 2.4em track, and its own internal 3 columns overflow/collapse into that width. desktop-03-boards-7.png shows exactly this: "theeoge#h0ly3jdailies" — a real overlap smear, not embellishment. mobile-03-boards-7.png shows the same root cause manifesting as ugly wrapping instead of overlap (narrower viewport), which the P0 fix also happens to resolve.
- Edge-fade not scroll-aware: styles.css:2298-2302 applies a static right-edge `mask-image` unconditionally at ≤640px, not gated on scroll position. mobile-03-boards-lifetime.png confirms: "Lifetime," fully scrolled into view as the last/selected segment, still renders with its right edge faded as if clipped.
- Sub-24px target: `.trend-row-toggle` has `min-height: 0; padding: 0;` (880, 883) with 0.88rem font — well under the 24px WCAG 2.5.8 floor the finding cites.
- Incomplete tablist: role="tablist"/"tab"/aria-selected present (Boards.tsx:287-303) with no aria-controls, no tabpanel, no roving tabindex/arrow-key handling — an accurate "incomplete" characterization.
- CTA-before-board: mobile-03-boards-today.png shows the CTA directly under the pair title, above the ranked list; mockup-boards-trends.png shows leaderboard → DNF → "Paths hidden" → CTA at the very bottom. Spec prose (§Boards, Today/Yesterday paragraph) also lists the CTA last in its enumeration, reinforcing the mockup.
- Scope/risk: confirmed via grep that `onRaceChallenge` only wires to `openRacePreviewFor` (preview, not run-creation) — the CTA reorder can't violate invariant 3. Files touched (Boards.tsx, styles.css) are correctly scoped; nothing here touches race timing, run lifecycle, App.tsx routing latches, or server queries. No spec invariant is at risk.

GAPS (why amend, not keep):
1. The ARIA fix is left as an "or" (complete the tablist vs. drop tab roles), but the two paths have very different test blast radii that the package doesn't surface. App.test.tsx has 12 call sites using `getByRole("tab", {name: ...})` against these exact buttons (lines 3751, 3766, 3773, 3887, 3932, 3944, 3970, 3999, 4038, 4072, 4099, 4122). Dropping tab roles breaks all 12 and requires rewriting them to `getByRole("button", ...)`. Keeping role="tab" and adding roving-tabindex/arrow-key handling avoids that breakage but adds genuinely new interactive behavior with zero existing coverage — the "missing test requirement" the review is supposed to flag.
2. Similarly, Open Question 1 (fit all 5 segments vs. move Lifetime behind 30d) has divergent test cost the package doesn't scope: App.test.tsx:3743-3752 hard-asserts the exact 5-tab array including "Lifetime," and the Bug-B regression test at 3754-3778 specifically taps "Lifetime" as a top-level tab. Moving Lifetime behind 30d would require rewriting both, plus new UI for the demoted control — real L-effort work bundled inside an M estimate.
3. Feasibility of "fit all 5 via the narrow-width pattern": the only existing precedent (`.mode-nav button` at styles.css:2518-2523) fits 4 short labels via padding/font shrink alone. Boards needs 5 segments including 9-char "Yesterday" and "Lifetime" — copy-paste of that pattern is not guaranteed to clear 320-375px, which is exactly the package's own "must hold at 320px" risk note, but it's stated as a caveat rather than a planning input.
4. Acceptance criterion #4 ("CTA after the board content") is ambiguous about whether "board content" includes the DNF section and the "Paths hidden" footnote. The mockup places the CTA after both, dead last. As worded, an implementer could plausibly place the CTA between placements and DNF and technically satisfy the criterion while missing the ratified mockup's order.
