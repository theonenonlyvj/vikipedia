## Changes
(1) Hierarchy: give 'Try again'/'Play again' the coral `.start-race-button` class and 'View leaderboard' the existing `.secondary-button` (styles.css:296-299) — verified both are currently bare `<button>`s (RaceResults.tsx:141-152). (2) Routing: App.tsx:1316 unconditionally sends 'View leaderboard' to global Boards even after a custom-challenge race; route through the existing `exitCompletedRaceToChallenge(challenge.id)` (App.tsx:845-850, already used by onOpenChallenge one prop over) whenever the raced challenge isn't today's actual daily. (3) Share: move ShareResultButton from dead-last to directly under the result header with a distinct treatment, and un-gate both ShareResultButton and ClaimCta from `status === "completed"` (RaceResults.tsx:172-187) so DNF runs can share ('DNF · 0:04 · 1 clk — beat that') and guests can still claim after a loss. (4) Surface: return `.result-panel` (styles.css:1444-1453) from the one-off coral-tinted wash to the standard --ink-raised/--line card, keeping coral as a left-border accent like `.daily-hero`. (5) Copy/exits: first-use expansion 'DNF — Did not finish' in DnfResultHeader; demote PlayAnotherCard to the mockup's smaller bordered secondary card; add a low-emphasis Home text link so the daily loop can close on Home.

## Acceptance criteria
- DNF screenshot shows exactly one coral primary (Try again), a secondary-styled View leaderboard, a Share affordance directly under the header, a claim CTA for guests, and 'DNF — Did not finish' spelled out
- After racing a non-daily challenge, 'View leaderboard' lands on that challenge's own board, not the daily Boards
- Results panel background matches the app's standard card surface (no warm/maroon tint)

## Risk
The routing change touches App.tsx's exit state machine — regression-test daily, custom-challenge, and DNF exits; share-on-DNF needs share-text copy that doesn't read as bragging about a loss.

## Judge A (amend) — design integrity
AMENDMENTS: 1. Add explicit DNF-awareness to composeShareText/ShareResultButton (src/race/shared.tsx:98-107, 115-137). Right now composeShareText only branches on `rank !== null`; a DNF has no rank, so un-gating ShareResultButton for DNF as currently written would produce "VWiki Race — <label> — 0:04 · 1 clk — <url>" with no "DNF" marker — reads exactly like a blazing-fast win, not a failed 1-click abandon. The package's own acceptance example ("DNF · 0:04 · 1 clk — beat that") requires a status/DNF flag threaded into composeShareText that the "changes" section doesn't call out — add it explicitly so whoever implements doesn't ship a misleading share string.

2. Soften item (5)'s Home-link framing. The covers-list claim "the loop the whole product is built around doesn't close on Home" overstates it against the app's own ratified north star, which defines the ritual as "races the daily, checks the board, leaves" — not "returns to Home." View leaderboard already IS the designed next beat post-race, and it (like Browse) already round-trips through AppShell/mode-nav where Home is one tap away. Keep the low-emphasis Home text link as a nice-to-have polish item, but don't treat it as load-bearing or put it in the acceptance criteria as if the ritual is currently broken without it — it isn't.

3. Note explicitly that PlayAnotherCard is a shared component (src/components/PlayAnotherCard.tsx, per its own doc comment "Home's post-play card... and Results' Play-another slot... share this one component"). Demoting its suggestion button to a smaller/bordered style (per mockup panel 3) will also restyle Home's "Got a few more minutes?" card, not just Results'. That's actually a second consistency win worth keeping (Home's live suggestion button is currently also bare-default-cyan, same problem) — call it out as an intentional twofer in the package rather than a side effect someone discovers mid-implementation.

4. Minor: "onOpenChallenge one prop over" (App.tsx) is loose — onOpenChallenge is at line 1307 and onShowLeaderboard at 1316, with 8 other props between them. Not wrong, just imprecise; doesn't change the fix.

5. Optional strengthening, not required: styles.css:674-677 (`.error-banner`) uses the identical coral-wash formula (rgba(255,125,102) border/background) as `.result-panel`. Worth citing alongside the ".daily-hero" precedent — it shows the coral wash is borrowed from the app's *error* semantics, so today a race win is presented in the same visual register as a warning banner. Strengthens the case for change (4) beyond "one-off," and explains why it reads wrong even to a first-time viewer.

## Judge B (amend) — engineering risk
AMENDMENTS: 1. TEST GAP (will actually break, not just "needs regression testing"): App.test.tsx:1030-1064, test "refreshes the completed challenge leaderboard exactly once after acceptance", explicitly races a non-daily fixture challenge (comment at :1042-1043: "This fixture challenge isn't flagged as today's daily") then asserts clicking View leaderboard lands on Boards' Today (:1059-1060) — it only passes today because Boards' `todaysHeroChallenge` fallback (Boards.tsx:63-68, "daily-or-fallback... same fallback Home's hero uses") happens to coincide with the raced fixture in this catalog, not because routing is challenge-aware. Once PKG-05's non-daily routing lands, this exact test's assertions (`Boards` heading, `boardCalls(...,"challenge-0001")===2`) will fail. The package's "risk" line ("regression-test daily, custom-challenge, and DNF exits") is too generic — add explicit acceptance items: (a) rewrite/split this test into a genuine-daily case (still → Boards Today) and a non-daily case (→ Challenge Detail via exitCompletedRaceToChallenge), (b) add a DNF case asserting Share/Claim now render, (c) add a case asserting hierarchy classes (coral primary vs secondary) are applied.

2. composeShareText/ShareResultButton (race/shared.tsx:98-157) has no DNF branch. `composeShareText` only ever emits `#rank · time·clicks` or `time·clicks` — un-gating it for DNF (per "changes" item 3) will silently produce share text that reads identically to a completed run (no "DNF" marker), contradicting the package's own acceptance text ("DNF · 0:04 · 1 clk — beat that"). Amend "changes" item 3 to explicitly require a signature/logic change in shared.tsx (e.g., an outcome-status or dnf:boolean param threaded through composeShareText and ShareResultButton) — not just removing RaceResults.tsx's `status === "completed"` render gate. shared.tsx is already in the files list so this is in-scope, just needs to be named as a required code change, not implied.

3. isDailyToday routing decision: App.tsx does not currently import `dailyDateForChallenge` (only RaceResults.tsx does, confirmed via grep) and the `onShowLeaderboard` handler at App.tsx:1316 takes no arguments. Implementing "route non-daily to exitCompletedRaceToChallenge" requires App.tsx to independently compute isDailyToday from `race.challenge` (completed case) or `dnfResult.challenge` (DNF case, both confirmed in-scope/populated at click time before exitCompletedRaceTo's reset runs) against `currentCentralDate` (already held in App.tsx state). Call this out explicitly as an implementation step so it isn't duplicated/drifted from RaceResults.tsx's own identical isDailyToday calc at RaceResults.tsx:119 — ideally factor into one shared helper call site.

None of these are blocking: I verified all cited line numbers/claims are accurate (RaceResults.tsx:141-152 bare buttons, App.tsx:845-850 and :1316, styles.css:296-299 .secondary-button, styles.css:1110-1122 .start-race-button, styles.css:1431-1453 .result-panel sharing a base rule with --ink-raised/--line that the coral tint currently overrides, styles.css:752-754 .daily-hero's left-border-only coral pattern as the cited precedent).
