This is the core 'buttons/look-and-feel' consistency package the owner asked for. (4) Restore the clip-path motif on `.mode-nav button` (remove `clip-path:none` at 649, or a smaller cut sized for 44-48px buttons) and verify the ≤350px compaction breakpoint; give inactive nav buttons a real hover affordance instead of zeroing box-shadow/transform. (6) Header lockup: drop or replace the mini 'VWiki' `.viota-mark` stacked above the 'VWiki Race' h1 (AppShell.tsx:173-175) — mockups show a single wordmark. (7) Wrap Challenge Detail's leaderboard+history in the shared card chrome (`.leaderboard-panel` group at styles.css:1431-1442) so it stops being the only mode with no frame. (8) While editing these rules, add `--muted-label:#9fb8bd` and `--text-bright:#dffbfb` tokens and point the ~35 raw-hex occurrences of #eef7f8/#9fb8bd/#dffbfb at var() tokens.

## Acceptance criteria
- Coral appears only on race-action buttons (Race / Start race / End Run / Try again) in a full screenshot sweep of all modes at 390px and 1440px; active nav/segment/auth tabs are teal-treated
- Browse card titles render white with muted-grey meta; only the arrow/state accents are colored

## Risk

## Judge A (amend) — design integrity
AMENDMENTS: 1. REVERSE THE CORE FIX DIRECTION (item 2 in "changes" — this is the package's namesake move and its P0 justification, so getting it backwards matters). Do NOT add `.start-race-button`/coral to ChallengeDetail's "Race this." The evidence points the other way:
   - Spec text (docs/superpowers/specs/2026-07-18-ux-redesign-modes-design.md:254-259, RESOLVED by Vijay): "coral reserved for primary/destructive race actions: Start, End Run." Opening a preview is explicitly non-committal (invariant 3, spec:57-59) — it is not "Start."
   - 4 of 5 relevant mockups render the "open preview" CTA teal, not coral: mockup-home-stateful-v2 (Home's "▶ Race" pill, before-you've-played state), mockup-boards-trends ("▶ Race today's daily"), mockup-browse-detail ("▶ Race this"), mockup-race-flow-v3 (Play-another's "Race ›" on Results). Only mockup-onboarding-v2 — one early draft still stamped "Colors/style still TBD" like all the others — shows Home's Race button coral, and it's the outlier.
   - Code confirms Home's hero button (Home.tsx:228, `.start-race-button`, calls `onRaceChallenge`) and Boards' "Race today's daily" (Boards.tsx:415-422, same class, same prop) both resolve to `openRacePreviewFor` (App.tsx:660-673, comment: "Home's DAILY hero and Challenge Detail's 'Race this' both resolve here"), the identical non-committal action ChallengeDetail's teal button already performs. PreRacePreview's actual clock-starting button (PreRacePreview.tsx:75-80, "Start race ▶") is correctly `.start-race-button`/coral, and RaceMode's `.end-run-button` (styles.css:444-449) is correctly coral too.
   So the real bug is that Home's hero and Boards' CTA are wrongly coral, not that Detail's is wrongly teal. Fix: strip `.start-race-button` from Home.tsx:228 and Boards.tsx:415-422 (give them a teal/plain treatment instead, matching Detail/PlayAnotherCard/Browse, which need no change here), and reserve `.start-race-button` exclusively for PreRacePreview's Start and RaceMode's End Run. This is a smaller diff than the package proposes and is the direction that's actually spec-compliant — implementing the package as written would spread coral further and dilute the exact signal it claims to protect.

2. Retarget `.daily-badge`'s fix color. The package prescribes "teal-outline pill." mockup-home-stateful-v2 and mockup-onboarding-v2 both show the "DAILY · WEIRD · #7" badge as a neutral/muted grey informational pill — not teal. Teal in this system reads as an interactive/link signal; the daily badge is a status label, not a control. Retarget to a neutral grey (e.g. `var(--muted)` text on `var(--ink-soft)`/`var(--line)` border), not teal.

3. Narrow the mode-nav active-state claim. The package cites mockup-boards-trends and mockup-onboarding-v2 as evidence for a "teal outline/underline" mode-nav treatment, but neither mockup actually depicts the mode-nav in that state — boards-trends shows the segment control, onboarding-v2 shows the auth-mode tabs (both different components, and those two specific coral→teal-outline fixes ARE well-evidenced — keep them as specified). Every mockup that does show the mode-nav itself (home-stateful-v2, boards-trends, onboarding-v2, browse-detail) renders its active item as plain bold/white text with no pill, fill, or underline chrome at all. There's no mockup evidence for a "teal pill" nav; the ratified target may be a bigger change (drop the filled-chip treatment entirely for nav) than a color swap. Ship the clip-path/motif restoration and hover-state fix as specified (those are solid, code+screenshot-confirmed), but flag the active-state color choice as an open question for whoever builds it rather than prescribing teal.

4. Minor evidentiary footnote, not a blocker: the "Guest tab / 'Continue as guest' button read as duplicate primary actions" claim is a legitimate inference from code (App.tsx:1492-1555 — the Guest tab and the Continue-as-guest submit render together when authMode==="guest") but isn't directly shown in the supplied screenshots (both identity-dialog shots capture "Create New" active, not "Guest"). Keep the finding, just don't cite it as screenshot-confirmed.

Everything else in the package holds up under verification and should ship as specified: the Browse coral-wash CSS-specificity bug (styles.css:1704 `.challenge-list span` (0,1,1) beating `.browse-card-title-row` (0,1,0) at styles.css:1733 — confirmed both in code and in mobile-04-challenges.png/desktop-04-challenges.png, where every title/meta line renders salmon), the mode-nav clip-path removal and weak hover (styles.css:649,652-655), the auth-mode-switch and board-segment-control active-state coral→teal-outline swaps (styles.css:261-265, 847-851 — well-matched to mockup-onboarding-v2 and mockup-boards-trends respectively), the duplicated header wordmark (AppShell.tsx:173-175, confirmed against every mockup's single-wordmark-plus-avatar-glyph header), ChallengeDetail's missing card chrome (ChallengeDetail.tsx has zero wrapping `.leaderboard-panel`-style container versus every sibling screen), and the raw-hex/duplicate-muted-gray token cleanup (grep confirms `--muted:#a8c2c6` coexists with 12 raw uses of a different gray `#9fb8bd` and 26 raw uses of `#eef7f8`/`#dffbfb` bypassing existing tokens).

REASONING: I verified the package against the code, the live screenshots, and — critically — the ratified mockups it cites as evidence, and against the design spec's explicit color rule.

Most of the package holds up extremely well: the Browse coral-wash bug is a genuine, cleanly-diagnosed CSS specificity defect (`.challenge-list span` at styles.css:1704 has higher specificity than the `.browse-card-title-row` reset at styles.css:1733, confirmed both in the CSS and visually in mobile-04-challenges.png where every title/meta line is salmon-colored) — this alone justifies shipping the package. The mode-nav clip-path removal, the header's duplicated wordmark, ChallengeDetail's missing card chrome, the auth-mode-switch/board-segment-control coral-to-teal-outline swaps, and the token-hygiene item are all independently verified against code and screenshots and are sound, low-risk, well-targeted fixes.

But the package's headline move — its namesake fix, stated as P0 justification — is backwards. It proposes making ChallengeDetail's "Race this" coral to match Home/Boards, framing Detail as the outlier. Tracing the actual click handlers (App.tsx:660-673) shows Home's hero, Boards' CTA, and Detail's "Race this" all resolve to the exact same non-committal `openRacePreviewFor` — none of them start the clock. The spec explicitly resolved "coral reserved for primary/destructive race actions: Start, End Run" (design doc line 254-259), and 4 of the 5 mockups that show this exact button (home-stateful-v2, boards-trends, browse-detail, race-flow-v3) all render it teal — only one early, equally-TBD-flagged mockup variant (onboarding-v2) shows it coral, and that's the one the live Home/Boards implementation happens to follow. So the defect is that Home and Boards are wrongly coral, not that Detail is wrongly teal. Shipping the package's fix as written would spread coral to more surfaces and dilute the exact "coral = commit to the clock" signal the package claims to be restoring — a real regression risk the judge brief asked me to watch for. I also found the `.daily-badge` target color (teal) doesn't match the mockups (which show neutral grey), and that the mode-nav active-state fix cites mockups that don't actually depict the mode-nav's active state (they depict different, similarly-named components).

None of this invalidates the package's diagnosis of a real, confirmed problem (functionally-identical CTAs rendering in different colors across screens) — it just means the specific remedy needs to be corrected before it ships. That's an amend, not a keep-as-is or a drop.

## Judge B (amend) — engineering risk
AMENDMENTS: 1) Add a test guard for the exact bug being fixed: an RTL assertion (in App.test.tsx, near the existing "Detail's Race this button" tests at ~2573) that ChallengeDetail's race button carries `start-race-button` — the same class Home.tsx:228 / Boards.tsx:416 / PreRacePreview.tsx:75 already use — so a future edit can't silently reintroduce the two-color-CTA bug. Currently the package has zero automated test requirements; everything in "acceptance" is manual/visual. That's acceptable for the pure-CSS items but not for the two code-touching ones (className add, card-chrome wrap), which are trivially testable and the repo's test culture (App.test.tsx is huge, RTL-heavy) supports it.

2) Fix the token plan (item 8) before executing it: styles.css already declares `--text: #eef7f8` (line 13) and `--muted: #a8c2c6` (line 14) — both are currently unused anywhere in the file (verified: zero `var(--text)` / `var(--muted)` references). The package's plan to mint two new tokens (`--muted-label`, `--text-bright`) without addressing these two dead ones will leave the file with 4 grey/white tokens instead of consolidating to 2. Amend: point the ~10 `#eef7f8` occurrences at the existing (currently-dead) `--text` token rather than a new one; decide whether `--muted` (`#a8c2c6`) or the new `--muted-label` (`#9fb8bd`) is the canonical "muted grey" — these are two different hex values, not duplicates of the same token — and retire whichever loses.

3) Coverage bullet "Guest tab and 'Continue as guest' button look like duplicate primary actions" isn't fully closed by the described changes. Verified: the guest-mode submit button (App.tsx:1550-1555, "Continue as guest") is a bare `<button type="submit">` with no className, so it already renders in the *default* base-button teal (styles.css:50-64), not coral — it's not actually a same-hue duplicate of the coral active tab today. The plan (item 3) only restyles the tab (`.auth-mode-switch button[aria-pressed=true]`, 261-265) to teal-outline and never touches the submit button, so post-fix both end up teal (one outline, one fill) rather than resolving whatever "duplicate primary" meant visually. Either drop this bullet from scope with a one-line note (defer to a follow-up), or add an explicit acceptance line for this specific pairing so it isn't silently claimed as fixed by the broader "full screenshot sweep" criterion.

4) Minor completeness note on the clip-path restore (item 4): the risk callout only mentions the ≤350px mobile end. Confirmed the mode-nav button is also a different geometry at desktop (≥880px, styles.css:2559-2563: min-height 40px, row layout, 8px/14px padding) vs. mobile's 48px square buttons — removing `clip-path:none` at line 649 falls back to the base button's 8px cut (line 63) at *both* ends since no media query overrides clip-path. That's probably fine (8px is already the smallest cut used anywhere), but the acceptance criteria/visual sweep should explicitly check the cut at the 40px desktop nav height too, not just the narrow-mobile compaction breakpoint.
