## Judge B (amend) — engineering risk
AMENDMENTS: 1. STAGE 1(b) IMPLEMENTATION GAP — spell out the JSX restructure, not just a CSS extraction. Home's docking pattern works because `<div className="daily-hero challenge-route">` (Home.tsx:194) wraps `.daily-hero-copy` and the `.player-gate` CTA as two grid-item siblings inside ONE container, which is what lets `grid-template-columns: minmax(0,1fr) auto` (styles.css:2592-2596) dock them side by side at ≥880px. Boards.tsx and ChallengeDetail.tsx do NOT have this shape today: Boards renders `<div className="board-segment-header challenge-route">…</div>` and then, as a separate sibling below it, `{showRaceCta ? <div className="player-gate">…</div> : null}` (Boards.tsx:402-424); ChallengeDetail is the same pattern (ChallengeDetail.tsx:46-63). "Extract into a shared `.route-header` rule and apply it" undersells this — merely adding a class to the existing two-sibling-section markup will not dock anything; the JSX in both files needs to be restructured so the title block and the CTA button become grid-item children of one wrapping element, mirroring Home's structure. Worth being explicit about this so whoever implements doesn't ship a no-op CSS class and re-report the same finding as unfixed. (Verified by reading Home.tsx:192-237, Boards.tsx:400-424, ChallengeDetail.tsx:40-63, and styles.css:752-770/2592-2601.)

2. STAT-GRID FIX — drop "auto-fit minmax(160px,1fr)" as an option; it does not reliably fix a fixed-count-6 grid. `.stat-grid` is only consumed by You.tsx (6 `<dt>/<dd>` tiles, confirmed via `grep -rn "stat-grid" --include=*.tsx`), and auto-fit's column count is a function of container width, not item count — at some intermediate width auto-fit could just as easily produce 5 columns (row of 5 + row of 1) as it could 3 or 2. `repeat(3, minmax(0,1fr))` (or `repeat(2,…)` to match the existing ≤980px override at styles.css:2256-2258) is the only one of the two proposed options that deterministically closes every row for exactly 6 tiles — spec the fix as that, not "3 or auto-fit."

3. MINOR FACTUAL FIX in Stage 1(a) — `.app-shell` already has `min-height: 100vh` (styles.css:101); it's just a block `<main>`, not a flex container, so the footer never gets pushed down. The missing piece is `display:flex; flex-direction:column` (+ `margin-top:auto` or equivalent on `.site-footer`), not `min-height`. Correct the changes text so the implementer doesn't waste a step re-adding a rule that's already there. (Also confirmed no regression risk here: `.race-takeover`, which shares the same `.app-shell` parent during a race, is `position:fixed; inset:0` — fixed-positioned elements escape flex-item sizing, so making `.app-shell` a flex column will not touch the race takeover's layout.)

4. SPLIT THE TICKET — Stage 1 is a same-day, zero-sign-off, CSS/JSX-only fix; Stage 2 is new desktop composition design work explicitly gated on the owner reviewing a screenshot before it rolls anywhere. Bundling them under one "L" package risks Stage 1's quick, low-risk win sitting blocked behind Stage 2's open design question. Recommend splitting into two packages so Stage 1 can ship and verify independently.

5. STAGE 2 VS. Stage 2's "real ≥880px compositions, content-left + rail-right" is a genuine reversal of that stated principle, not just an unspecified detail. The package already requires owner sign-off before rollout, which is right, but given the spec is described as a "council-ratified contract," recommend that sign-off take the form of a spec addendum (even a short one) rather than an informal screenshot approval, so the design doc stays the single source of truth instead of drifting out of sync with what actually shipped.
