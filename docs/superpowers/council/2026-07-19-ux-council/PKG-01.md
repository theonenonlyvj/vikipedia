## Changes
Kill the dual derivation in AppShell.tsx (~115-130): Boards' Today segment currently gets `selectDefaultChallenge` (which silently falls back to activeChallenges[0] — an arbitrary user challenge) while Home gets `selectHomeHeroChallenge` (honest yesterday-daily fallback with a `kind` field). Pass the same homeHero selection to Boards. When `kind !== "today-daily"`, Boards must render Home's honest framing ("YESTERDAY'S DAILY · <flavor>" badge, "New daily drops 5:00 AM Central") — never an unqualified "TODAY" label or "Race today's daily" CTA on a non-daily challenge. Make the activeChallenges[0] fallback unreachable from any nav surface: if no daily exists at all, show an explicit empty state. Delete the now-false comment at Boards.tsx:63-66 ("the two screens can never disagree"). Then add the spec-mandated pinned daily row at the top of Browse ("⭐ DAILY #N · pair" per mockup-browse-detail), sourced from the SAME homeHero selection, routing to Home via onSelectMode("home") — above "ALL CHALLENGES", visually distinct from the catalog.

## Acceptance criteria
- With no daily generated today (current prod state), Home, Boards→Today, and the Browse pinned row all show the same challenge pair with the same 'Yesterday's daily' framing — screenshot at 390px and 1440px
- Boards' Today tab can never display a challenge that carries no daily badge in Browse; the '▶ Race today's daily' label only appears when the shown challenge is actually today's daily
- Browse shows a pinned daily row above the catalog that navigates to Home, not Challenge Detail

## Risk
Boards' Today leaderboard fetch is keyed by challenge id — confirm the board query follows the new selection in both healthy-daily and fallback states, or Boards will show the right title over the wrong board.

## Judge A (amend) — design integrity
AMENDMENTS: 1. Make the Today-tab pre-drop behavior an explicit decision in the package, not an implicit "just mirror Home" instruction. Two candidate resolutions, either is fine, but pick one and say so:
   (a) Mirror Home's yesterday-daily framing on Today too (package's current implicit choice) — accept that Today and Yesterday will show the same board pre-drop, and add a one-line rationale for why that duplication is acceptable/intentional; or
   (b) Have Today render its own honest empty state pre-drop ("No daily yet — next one drops 5:00 AM Central · see Yesterday's ›" linking to the Yesterday tab) and leave Yesterday as the sole holder of yesterday's board. This avoids two tabs rendering identical content and keeps the Today/Yesterday segmented control's semantics intact.
   Either way, the CTA copy must stop saying "Race today's daily" whenever `kind !== "today-daily"` — package already says this, keep it, but make sure whichever framing option (a or b) is chosen carries a CTA label review to match (Home already downgrades to a bare "▶ Race" in this state — mobile-01-home).

2. Effort label: reclassify the Browse pinned-daily-card portion as its own line item at effort L, not folded into the overall M. It's a new, visually-distinct UI element (⭐ badge, "done" state, routing branch to `onSelectMode("home")` instead of `onOpenChallenge`) per mockup-browse-detail, on top of the AppShell/Boards wiring fix, which is the actual M-sized part. Splitting the estimate doesn't change scope, just makes sequencing/staffing honest — the wiring fix (AppShell + Boards) is safe to ship first and independently resolves the P0 "random challenge mislabeled TODAY" bug even before the Browse card lands.

3. Acceptance criteria should add one more line to close the gap this review found: "Boards' Today tab and Yesterday tab never render byte-identical board content under different headers" (or, if option (a) above is chosen instead, explicitly accept and document the duplication rather than leaving it as an unstated side effect).

REASONING: Verified the core claim directly against code and prod screenshots, and it holds exactly as stated.

Code: AppShell.tsx:115-121 computes `todaysHeroChallenge` via `selectDefaultChallenge` (challengeSelection.ts:38-53, whose fallback chain ends at `activeChallenges[0]` — an arbitrary catalog entry) and passes it to Boards, while Home gets the separately-computed `homeHero` from `selectHomeHeroChallenge` (challengeSelection.ts:78-95), which has an honest yesterday-daily fallback with a `kind` tag. Boards.tsx:63-66's doc comment ("so the two screens can never disagree about which challenge is 'today's'") is demonstrably false today — confirmed exact line numbers.

Screenshots prove this isn't theoretical: mobile-01-home shows Home's honest "YESTERDAY'S DAILY · WEIRD / Coffee -> Great Molasses Flood / New daily drops 5:00 AM Central." mobile-03-boards-today simultaneously shows "TODAY / Moon -> Gravity" with a "▶ Race today's daily" CTA. Cross-checked against mobile-04-challenges: "Moon -> Gravity" is Challenge #1, created by theonenonlyvj, carrying no DAILY badge at all — it's `activeChallenges[0]`, not a daily. Boards is actively lying about what's racing "today." mobile-03-boards-yesterday, by contrast, correctly matches Home ("Coffee -> Great Molasses Flood"), confirming the bug is specifically in the Today-tab derivation, exactly as the package states.

Spec-fidelity claim on Browse also checks out: the design spec (line 185-186) explicitly says "The daily pinned at top but pointing to Home," and mockup-browse-detail shows a "⭐ DAILY #7 · Coffee → Great Molas… ✓done" row above "ALL CHALLENGES." Live mobile-04-challenges has no such pinned row — dailies are just interleaved into the flat numbered catalog. Confirmed gap, not invented scope; it's explicitly gated to Increment 5 ("Challenges/Browse full card spec"), the increment this codebase is currently in.

This is squarely what the owner flagged ("screens still aren't consistent") — not scope creep, and not a taste downgrade (the fix reuses Home's already-shipped honest copy voice rather than flattening it). The risk note about the board-fetch query is well-founded but self-resolving: Boards.tsx:156 keys `getChallengeBoard` off `activeChallenge.id`, which already tracks whichever challenge object is passed in as `todaysHeroChallenge` — so re-pointing that prop to `homeHero.challenge` in AppShell automatically carries the board fetch with it, confirmed by reading the effect.

One real gap in the proposal: it doesn't resolve what Boards' *Today* tab should show pre-drop, it just says "render Home's honest framing." I checked the spec's Boards section (no pre-drop case defined there — the spec assumes today's daily always exists) and the mockup (no pre-drop Boards state shown either). Silently copying Home's yesterday-fallback onto the Today tab would make Today and Yesterday tabs render near-identical boards under different headers (Boards already has a dedicated Yesterday tab a few taps away) — a real redundancy the package doesn't acknowledge or choose between.
