## Changes
(1) How-to-play placement: TeachingGate.tsx:56 borrows the `identity-dialog` class for base styling, so the popup matches `.modal-backdrop:has(.identity-dialog)` (styles.css:2420) — a top-anchor rule built solely for the iOS Safari keyboard bug in the save-stats dialog — producing the top-pinned card with a half-screen dead zone on mobile. Remove the shared class (share visual rules via a selector group `.identity-dialog, .teaching-gate-dialog { … }`) or rescope the :has() to a new `.save-stats-dialog` class only the real dialog carries; How-to-play falls back to the default centered/bottom-sheet placement End Run already uses. While there, give `.teaching-gate-dialog` real rules (heading ~1.8rem like `.recovery-notice h2`, tighter padding, softer shadow) so a 3-line tip stops wearing signup-form chrome. (2) Modality: when any ModalDialog opens, set `inert` on the sibling `.shell-topbar`/`.content-shell`/`.site-footer` so virtual-cursor AT can't wander into the background; clear on close. (3) Focus: on RaceResults mount, move focus to the result heading (tabIndex=-1 + ref.focus()), covering the DNF case where nothing currently receives focus. (4) Focus ring: replace the near-invisible `.name-control input:focus` outline (rgba cyan @0.22, ~1.7:1) with the app's strong :focus-visible ring or a solid ≥3:1 equivalent. (5) Targets: min-height 44px for `.auth-mode-switch button` and identity-form buttons (simplest: `.identity-dialog button { min-height:44px }`), and make `.target-reference summary`'s 44px the base rule, not mobile-only. (6) In App.tsx handleArticleClick (~1237-1253), bail out of preventDefault when ctrlKey/metaKey/shiftKey or button!==0 so modifier/middle-click behaves natively.

## Acceptance criteria
- Mobile How-to-play renders centered or as a bottom sheet with no half-viewport dead zone, and visibly lighter-weight than the identity dialog; desktop unchanged
- With any dialog open, screen-reader virtual cursor cannot reach nav/footer items; after a DNF, focus lands on the result heading
- Identity dialog buttons measure ≥44px; focused text inputs show a ≥3:1 indicator; cmd/ctrl-click on an in-article link opens a new tab
- The original iOS save-stats keyboard fix still works (regression-test the password field on iOS Safari)

## Risk

## Judge A (amend) — design integrity
AMENDMENTS: 1) Re-verify the "identity/onboarding buttons below 44px on mobile" claim before implementing — as written, styles.css:2324 already has a blanket `button { min-height: 44px }` rule inside `@media (max-width: 640px)` that covers ALL `<button>` elements including `.auth-mode-switch button` and `.identity-form button` (App.tsx:1466-1656 confirms they're plain `<button>` tags, no class overriding min-height). My reading of the cascade says these are already ≥44px on mobile. If a real gap exists it's more likely on desktop/tablet widths (>640px, base rule is only 38px at styles.css:57) where the identity dialog is equally reachable (e.g. touch laptops) — restate the finding to target that width range, or drop it if re-measurement on-device shows mobile is already fine. Don't ship a no-op CSS change presented as a fix. (The paired target-reference 44px claim, by contrast, is correctly verified — its own 44px rule only lives inside the mobile media query at styles.css:2340-2341, base is 36px at :551-553 — keep that one as-is.)

2) Fix or drop the modifier/middle-click item as currently scoped. The in-article anchors' `href` is rewritten to an internal synthetic hash (`#article:<title>`, services/wikipediaSanitizer.ts:267), and App.tsx has no hashchange/location.hash handling anywhere (grepped, zero matches) — there's no hash router. So bailing preventDefault on ctrl/cmd/shift-click will make the browser open a new tab at `<origin>/#article:Title`, which the SPA will simply ignore and render its default landing route — not the article the user clicked, and not race-in-progress context (which can't exist in a fresh tab anyway). That's a confusing "looks broken" outcome, arguably worse than today's silent block. Two better paths: (a) point the anchor's real `href` at the actual external Wikipedia URL already captured in `data-vwiki-race-href` (wikipediaSanitizer.ts:269) so modifier-click opens the genuine Wikipedia article in a new tab — this is probably the actually-useful, on-brand behavior (peek at the source without losing your run) — or (b) drop the item. Also soften the "middle-click" framing: modern browsers fire `auxclick`, not `click`, for the middle button on an `<a>`, so React's onClick (and this preventDefault) likely never sees a true middle-click in the first place — verify before keeping that half of the claim.

3) The inert/aria-hidden-background and RaceResults-focus-on-mount items are both verified real (confirmed no aria-hidden/inert anywhere in App.tsx/modes/components, confirmed RaceResults.tsx has zero focus management for either the completed or DNF outcome) and are legitimate hardening, but they are accessibility completeness fixes, not what the owner flagged (visual/brand consistency). Keep them bundled since they touch the same dialog code already being reworked for item 1 — don't let them inflate the package's stated priority/scope beyond "P1 dialog fix + drive-by hardening while the file's open."

The focus-ring-consistency finding is also directly on-brief ("look and feel... don't all feel consistent") and I confirmed the bug mechanism precisely: `.name-control input:focus` (styles.css:472-476, specificity 0,0,2,1) silently beats the app's global `input:focus-visible` white-ring rule (styles.css:82-88, specificity 0,0,1,1) via higher specificity, producing a fainter, inconsistent ring exactly as claimed.

Two sub-claims don't hold up under verification, which is why this is "amend" not "keep" outright: (a) the mobile 44px auth-button claim looks contradicted by the codebase's own blanket `button{min-height:44px}` mobile rule — the buttons are very likely already fine on the mobile viewports the finding cites; (b) the modifier/middle-click fix as specified would produce a new, confusing failure mode (dead internal hash opened in a fresh tab) because the app has no hash router, which the package's author evidently didn't check — a case of "fix the letter of an accessibility rule without checking it produces a useful result," which is exactly the kind of unverified claim I'm supposed to catch here.

None of this is a taste/brand downgrade — nothing flattens the viota identity or cargo-cults the mockup over a better live treatment; if anything, right-sizing the how-to-play popup to match its own spec's "quick-dismiss" intent is a taste improvement. The inert/focus-management items are legitimate but are accessibility hardening riding along with dialog work already required for item 1, not something the owner asked for — fine to keep bundled at low added cost, but shouldn't be oversold as core to "design integrity."

## Judge B (amend) — engineering risk

REASONING: Verified against the actual files. No conflict with any spec hard invariant — the spec (docs/superpowers/specs/2026-07-18-ux-redesign-modes-design.md) has zero mentions of focus/inert/modal/touch-target, so this package is entirely outside the ratified contract's territory. It doesn't touch race timing, run lifecycle, placement, or server queries. The App.tsx change (handleArticleClick) is a pure guard-clause addition before the existing preventDefault/followArticleLink call — normal left-clicks are untouched, so click-counting/scoring is unaffected for the primary path. Low core risk overall, but the implementation-as-specified has real gaps I could confirm from code:

1. Inert-siblings selector list is wrong for the case that matters most. `.shell-topbar`/`.content-shell`/`.site-footer` only exist in the DOM when AppShell is mounted (AppShell.tsx:172,207,282, returned via a bare Fragment so they flatten to direct children of `<main className="app-shell">`). Both real ModalDialog usages in App.tsx can open while RaceFlow — not AppShell — is mounted instead: the End Run dialog (App.tsx:1390) opens *only* during an active race, and the identity dialog (App.tsx:1454) is also reachable from RaceResults' "Claim your spot" CTA mid-raceEngaged. In that state the DOM sibling is RaceFlow's single `<div className="race-takeover">` (RaceFlow.tsx:257), not any of the three named classes — so a literal implementation of "set inert on .shell-topbar/.content-shell/.site-footer" is a no-op for exactly the End Run dialog, the one dialog that most needs backgrounding since it interrupts a live timed race. Fix: implement inert generically inside ModalDialog.tsx (walk `.modal-backdrop`'s parentElement.children, inert every non-`.modal-backdrop` sibling, restore on unmount) — all three call sites already share the same DOM parent, so this is centralized, covers RaceFlow and AppShell alike, and is less code than threading a flag through 3 files.

2. The "open in new tab" acceptance criterion won't actually work. Anchors get `href="#article:<Title>"` (same-page hash, wikipediaSanitizer.ts:267) for in-app SPA nav; the real Wikipedia URL only lives in `data-vwiki-race-href` (wikipediaSanitizer.ts:269). Skipping preventDefault on modifier/middle-click makes the browser follow that hash href, opening a new tab to the *app's own current URL* with a meaningless fragment — not to Wikipedia. Needs an explicit `window.open(link.dataset.vwikiRaceHref, "_blank", "noopener")` (still calling preventDefault) rather than a bare bail-out.

3. No test exists or is specified for this exact behavior. App.test.tsx's only ctrlKey/metaKey tests cover an unrelated feature (Cmd/Ctrl+F blocking). Given point 2, a real regression test is needed: assert window.open receives the real sourceUrl on modifier/middle-click, assert followArticleLink/click-count is NOT invoked, and assert this holds specifically during an active timed race (so a stray Cmd-click doesn't silently cost the player a move — arguably the actual point of the fix, but currently unstated as a scenario).

4. Competing focus effect on the "completed" results path was missed. RaceMode.tsx:166-170's `WikipediaArticlePanel` (also rendered by RaceResults for the completed case, RaceResults.tsx:190-199) already has its own mount-effect that scrolls/focuses the *article* heading. The new "focus the result heading" effect will race against it; only the DNF path is genuinely silent today as claimed. RaceMode.tsx should be added to the file list and a test should assert final focus target for the completed case, not just DNF.

Two smaller, non-blocking notes: the "auth-mode-switch buttons fall below 44px on mobile" framing is slightly wrong (the generic mobile `button{min-height:44px}` rule at styles.css:2323 already wins there via cascade order; the real gap is desktop/tablet widths where base `button` is 38px) — the proposed CSS fix still closes the real gap, just verify on a >640px touch device, not phone-width. And unconditionally bumping `.target-reference summary` to 44px will grow `.path-strip`'s sticky height ~8px on desktop too; re-check `.article-heading h2{scroll-margin-top:132px}` (styles.css:1279) still clears it post-navigation.

None of this touches race timing/scoring/placement/routing latches, so risk to the game's core invariants is low — the issues are all in the "will the fix actually do what the acceptance criteria claim" category, which is fixable without re-scoping the package.
