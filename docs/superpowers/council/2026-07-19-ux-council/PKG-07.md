## Changes
Every ratified mockup shows 'DAILY · WEIRD · #7' and '1:23 left today'; neither shipped. (1) Sequential daily number: derive N from the date-ordered daily list (NOT challenge ids — the in-race 'CHALLENGE #7' kicker is a global creation counter and misleading for dailies). Render 'DAILY #N' in the badge on Home's hero, Boards' Today, and the pre-race preview; lead share text with it ('VWiki Race — Daily #7 — …' in composeShareText, shared.tsx:98-107) the way Wordle leads with 'Wordle 942'. (2) Live countdown to the 5:00 AM Central drop on Home's hero pre-play and the pre-race preview, replacing the static 'New daily drops 5:00 AM Central.' sentence (spec Open Question 3, resolved per mockup-home-stateful-v2/mockup-target-preview). (3) Unify daily badge vocabulary: pair date and flavor everywhere a daily badge appears ('DAILY 7/17 · WEIRD') so Browse's date pills and Home/Boards' flavor words stop being two disconnected labels for the same object. (4) Add a streak tile (current + best if tracked) to You's Stats grid reusing the accountStats.dailyStreak value Home already fetches; add one How-to-play line establishing cadence: 'A new pair drops every day at 5:00 AM Central — keep your streak alive.'

## Acceptance criteria
- The same 'DAILY #N' appears on Home, Boards Today, Preview, and in the copied share text for the same daily
- Home pre-play shows a time-remaining readout that visibly decreases across a reload; the static drop-time sentence is gone
- You tab shows a streak tile; the How-to-play modal mentions the daily cadence

## Risk
Numbering must be stable and shared (server-derived or derived from the same daily list every client sees) or two players' share texts will disagree; countdown needs a Central-time source of truth to avoid local-timezone drift.

## Judge A (amend) — design integrity
AMENDMENTS: 1. FIX THE NUMBERING DERIVATION — the package's method is technically wrong as written. "changes" says derive Daily #N "from the date-ordered daily list," but Home.tsx's own code comment (src/modes/Home.tsx:94-96) states "The catalog only carries active challenges... a genuine 'yesterday's daily' is often simply absent once its day passes" — the client never has the full historical daily list, so counting through it would produce a number that drifts as old dailies age out of the catalog, and would disagree between two clients loaded at different times. Two fixes, pick one at plan time: (a) pure date arithmetic — days-since-a-fixed-launch-epoch, computed identically to the existing `dailyFlavorForCentralDate` pattern in src/domain/dailyEditorial.ts:71-85 (same file, same style, zero server change, zero drift risk, self-heals across a skipped day like today's known daily-gen failure); or (b) a server-computed sequential field added to daily_features, if the team wants numbering to survive a launch-epoch redefinition. Either way, add a server/data-requirements file to the package's "files" list — right now it lists none, despite this needing an explicit decision the way the design spec calls out every other new data need. Note the existing code deliberately avoided this exact fabrication risk (shared.tsx:90, "no fabricated 'daily #N' — the catalog has no such field yet") — that comment is direct on-record evidence the risk is real, not hypothetical.

2. NARROW THE VOCABULARY FIX (item 3) — "DAILY 7/17 · WEIRD" everywhere is not what the ratified mockups show. Checked mockup-home-stateful-v2, mockup-target-preview, and mockup-boards-trends directly: every one uses FLAVOR + sequential-number ("DAILY · WEIRD · #7" / "DAILY #7"), never calendar-date + flavor. Browse's existing dailyBadgeLabel ("DAILY 7/17", src/domain/challengeSelection.ts:28-35, confirmed live in mobile-04-challenges.png) serves a different, legitimate job — scanning a mixed chronological catalog of dailies and non-dailies where a relative date is more scannable than a bare "#42." Don't homogenize Browse's badge to match Home/Boards; instead add the sequential #N to every badge instance (Home, Boards, Preview, Browse's existing date badge, and the share text) so the number is the unifying thread, and let Browse keep its date suffix since it earns its keep there.

3. SOFTEN THE COUNTDOWN SCOPE — "live countdown" (top-level changes text, and the data-viz/gameplay finding phrasing) overstates what the package's own acceptance criterion actually requires: "a time-remaining readout that visibly decreases across a reload" — i.e., computed at render time, not necessarily a ticking setInterval. Recommend stating this explicitly in the package so implementation doesn't over-build a live ticking timer (extra re-render cost, more surface for the Central-time-drift risk already flagged) when a render-time computation satisfies the acceptance bar.

4. Effort is optimistic at M — numbering-scheme decision + tests, a countdown readout, vocabulary changes across 5 render sites, a You.tsx addition, and a TeachingGate copy change is close to the M/L boundary. Fine to keep at M if scoped per the fixes above, but flag it may want to split into two smaller PRs (numbering+countdown vs. streak-tile+copy) rather than one M-sized change.

## Judge B (amend) — engineering risk
AMENDMENTS: 1) Split the package. Keep as-is (low risk, ready to build): the You.tsx streak tile (reuses `accountStats.dailyStreak`, already fetched — see Home.tsx:307-320 for the exact field) and the one-line How-to-play cadence addition to TeachingGate.tsx. Descope/rework separately (see below): "DAILY #N" sequential numbering + badge-vocabulary unification + the countdown, which are not buildable as scoped.

2) Fix the numbering data source before estimating effort. There is no sequential-daily-number field anywhere in the type system (`Challenge`/`DailyFeature` — domain/types.ts:54-69, domain/dailyEditorial.ts:6-10) and `listChallenges` only returns `WHERE c.is_active = 1` (d1TrackingRepository.ts:829) — Home.tsx's own comment says "the catalog only carries active challenges" (Home.tsx:94-96) and Boards.tsx's says "a genuine 'yesterday's daily' is often simply absent once its day passes" (Boards.tsx:134-139). Client-side "date-ordered daily list" derivation (as the changes text proposes) is therefore already wrong today for any daily outside the active window, and will disagree between clients loaded at different catalog states — the exact failure the package's own "risk" field warns about, except it's not a future risk, it's true now. Add a server-computed ordinal (e.g. `ROW_NUMBER() OVER (ORDER BY daily_date)` against `daily_features`, d1/migrations/0005_editorial_dailies.sql:94-104) exposed via a new/extended endpoint, and add worker.ts / d1TrackingRepository.ts / contracts.ts / apiHandlers.ts / App.tsx to "files." This is genuinely L effort for the numbering sub-feature, not M — treat it as its own package or an Increment-0-style prerequisite, since it's also a new Data Requirement the ratified spec never lists.

3) Add src/race/RaceResults.tsx and src/App.tsx to "files." `ShareResultButton`/`composeShareText` (shared.tsx:98-157) are called from both Home.tsx:263 and RaceResults.tsx:180; the latter is the actual most-common share moment (fresh finish, not the persisted post-play card) and acceptance criterion 1 requires the number in "the copied share text" — unsatisfiable without it.

4) Reconcile the three different badge formats currently in play: change (1) proposes "DAILY #N"; change (3) proposes "DAILY 7/17 · WEIRD"; the ratified mockup (mockup-home-stateful-v2, mockup-target-preview) shows a third format, "DAILY · WEIRD · #7". Pick one and update both change items — as written the package gives the implementer three incompatible targets for the same string.

5) Name the specific existing tests this will break and require them to be updated as part of the package, not discovered later: App.test.tsx:3212 and :3456 assert `getByText(/new daily drops 5:00 am central\./i)` is visible on Home pre-play — directly contradicted by "replacing the static ... sentence"; :3485 asserts its absence in another state and needs re-checking too. Clarify whether RaceResults' separate "🔥 Day 1 · New daily drops 5:00 AM — come defend your spot" hook (RaceResults.tsx:137, tested at App.test.tsx:2899 and :2943) is in scope — it uses near-identical copy but is a different, first-finish-only trigger; the package should say explicitly which of the 3 near-duplicate strings survive vs. get replaced.

6) Add DST-safe test coverage as an explicit requirement for the countdown, not an afterthought. No timezone library exists in the repo (package.json has no date-fns-tz/luxon/dayjs); the only existing Central-time helper (challengeSelection.ts:9-10) produces a calendar-date string, not an instant-of-day countdown. This is a real, not hypothetical, hazard — the server's own daily-drop cron already fires twice at different UTC times specifically to land on 5:00 AM Central across DST (wrangler.api.toml:10: `crons = ["0 10 * * *", "0 11 * * *", ...]`). Land the countdown as a pure function in src/domain/ with its own *.test.ts covering both DST-transition days, matching this codebase's established pattern (challengeSelection.test.ts, dailyTrends.test.ts, teachingGate.test.ts) rather than inline component logic — none of Home/Boards/You/PreRacePreview/TeachingGate currently have dedicated test files, so new logic needs to go in a tested domain module, not more untested JSX.

7) Verify or drop the "in-race 'CHALLENGE #7' kicker" justification in change (1). It describes UI that does not exist anywhere in the shipped app (no match for "CHALLENGE #" or a creation-counter kicker in src/race or src/modes) nor in the ratified race-flow mockup (mockup-race-flow-v3's HUD shows only timer/clicks/Target/End Run; Results shows "#2 today" placement, never a creation-id kicker). The underlying recommendation (don't derive the daily number from challenge creation id) still stands on its own merits, but the cited evidence appears fabricated — worth a second pass on the rest of the package's factual claims before trusting the effort estimate.

REASONING: No hard spec invariant is violated: the package doesn't touch run creation, timing, DNF/completion precedence, placement dedup, identity gating, or path-hiding, and it doesn't touch App.tsx's routing latches or race-flow JSX that Increment 1 already isolated as the highest-merge-conflict surface — so there's no regression risk to the race engine itself, and no Increment-1/App.tsx merge-conflict exposure as scoped. That's real and worth keeping.

But two of the package's four changes are not implementable as scoped, and the effort/files estimate materially understates the work. The sequential "DAILY #N" numbering has no data source: neither the client-visible Challenge/DailyFeature types nor the `listChallenges` endpoint (is_active-filtered, confirmed at d1TrackingRepository.ts:829) carry it, and the code's own comments (Home.tsx, Boards.tsx) document that the client catalog already drops past dailies — meaning the package's proposed client-side derivation is wrong today, not just fragile later, which directly contradicts the "risk" field's framing of this as a future concern. The files list also omits RaceResults.tsx (the actual primary share surface) and any server file, even though satisfying the package's own acceptance criteria requires both. The badge format is specified three incompatible ways across the package's own changes and the ratified mockup. The plan to replace a static sentence with a countdown will break three named existing App.test.tsx assertions the package doesn't mention updating. And the countdown's DST correctness is a proven-nontrivial problem in this exact codebase (the server cron already double-fires across DST to hit "5am Central") with no library support and no domain-layer test plan.

None of this is a reason to drop the package — the underlying player-facing idea (daily numbering + countdown + streak visibility, all explicitly anticipated by the ratified spec's Open Question 3 and the mockups) is sound and low-risk on the engine side. But roughly half of it (numbering + badge unification + countdown) needs its data source, file scope, exact copy, and test plan fixed before an implementer could build it as written, and effort for that half should be L, not M. The other half (streak tile, how-to-play line) is genuinely ready at M/S effort. Recommend splitting and re-scoping rather than shipping the combined package as specified.
