## Changes
Five changes to the boards-rendering trio so every leaderboard surface tells the same truth. (1) Best-rank-per-account everywhere: Results' snippet (boardSnippetRowsFromLeaderboard, boardSnippet.ts:62-74) and Challenge Detail's LeaderboardList rank every attempt, so one account fills multiple slots ('#1 theonenonlyvj / #2 theonenonlyvj'). Collapse to the same best-per-account rule Home's getChallengeBoard already applies (client-side: group RankedLeaderboardRow by accountId, keep min rank, re-rank — but verify the tie-break matches the server rule). Repeat attempts live in 'Your history', not as extra board rows. (2) Anti-spoiler: LeaderboardList.tsx:50-63 renders 'View winning path' unconditionally, violating invariant 5; add a `pathsUnlocked` prop computed in ChallengeDetail from yourRows containing a non-abandoned run, showing Boards' 'Paths hidden until you've played' copy otherwise. (3) Delete the 'Server tracked' badge (it's the default, not information) and the 'Repeat run' badge (moot after dedup); keep only 'Historical' for pre-migration rows, with a tap-to-reveal explanation (details element) instead of a hover-only title attribute. (4) Add the `.is-you` + '(you)' self-row treatment to LeaderboardList, matching Boards/BoardSnippet. (5) Results header vs board row disagree on the just-finished run's time (client 0:04 vs server 0:05): once the server row lands, render the server-persisted elapsed in the header too — one source of truth.

## Acceptance criteria
- No display name appears at more than one rank on any board surface (Results snippet, Challenge Detail, Boards, Home snippet)
- Challenge Detail for a never-played challenge shows no path disclosures and the 'Paths hidden until you've played' copy; after completing it, paths appear
- No 'SERVER TRACKED' or 'REPEAT RUN' pills anywhere; viewer's own row is highlighted '(you)' on Challenge Detail
- The just-finished run shows the identical time in the Results header and its own board row

## Risk
Client-side dedup must match the server's best-run semantics exactly or Home and Detail will disagree in a new way — test with an account that has repeat runs including a better-later-run case.

## Judge B (amend) — engineering risk
AMENDMENTS: 1. REDIRECT THE DEDUP TO A SERVER QUERY, NOT CLIENT-SIDE GROUP BY. The codebase already has a proven, battle-tested best-rank-per-account query: `listChallengePlacements` (src/server/d1TrackingRepository.ts, ~line 2318+), exposed as `ChallengeBoardPlacement`/`ChallengeBoardDnfRow` (src/domain/types.ts:265-293, doc comment: "already invariant-2-correct... one row per canonical account"). It already powers Boards.tsx and Home via `getChallengeBoard`. The spec's own Increment 0 puts this dedup at the SQL layer specifically so every screen shares one source of truth ("GROUP BY account_id, MIN(rank)"). PKG-03 instead asks for a hand-rolled client-side re-derivation inside `boardSnippetRowsFromLeaderboard`/`LeaderboardList`, which the package's own "risk" field admits could diverge from server semantics — and it will, structurally: `listLeaderboard`'s SQL (d1TrackingRepository.ts ~2237-2300) ranks completions by `elapsed_ms ASC, click_count ASC, completed_at ASC` and DNFs by `click_count DESC, abandoned_at ASC`; faithfully re-implementing and keeping that tie-break in sync in TypeScript, forever, for zero benefit over reusing the existing query, is unnecessary regression surface.

2. BUT: reusing `listChallengePlacements`/`ChallengeBoardPlacement` as-is doesn't work either — it carries NO `runId` (types.ts:269, "Boards never discloses a per-run path this increment... nothing to key a path disclosure on") and no `protocolVersion`. ChallengeDetail (unlike Boards) is spec'd to keep per-row path disclosure after the viewer has played, and PKG-03 wants to keep a `protocolVersion`-driven "Historical" badge. Neither existing server query (`listLeaderboard` = per-attempt + runId + protocolVersion, no dedup; `listChallengePlacements` = deduped, no runId/protocolVersion) satisfies both needs at once. The real fix is a small server-side change: extend `listChallengePlacements` (or add a sibling query) to also return the representative row's `runId` and `protocolVersion` for the surviving best-rank-per-account attempt. This is genuine server work, not reflected in PKG-03's "files" list (which has zero server files) or its "M" effort estimate.

3. FILES LIST IS INCOMPLETE. Confirmed by tracing the actual data flow: `openChallengeDetail` in App.tsx (~line 644-658) fetches only via `refreshLeaderboard` (the raw `listLeaderboard`) — there is no existing board-endpoint fetch wired to Challenge Detail today. Whichever direction is chosen (new server query, or swapping to the board endpoint), App.tsx needs new fetch/state/prop wiring through AppShell → ChallengeDetail, and likely `src/services/vwikiRaceApiClient.ts` + `src/server/apiHandlers.ts` + `src/server/contracts.ts` for a new/extended endpoint shape. Add these files explicitly; effort is more realistically M/L, not M.

4. CHANGE (5)'s TIME-RECONCILIATION FIX ALSO TOUCHES App.tsx, NOT JUST RaceResults.tsx. Traced it: for DNF, `DnfResultSnapshot.elapsedMs` (App.tsx:1186-1193, `race.elapsedMs`) is captured from the client-side timer *before* `race.endRun()`/server confirmation, while the eventual board row's `elapsed_ms` is server-computed from `abandoned_at` (d1TrackingRepository.ts's CASE-based fallback in `listLeaderboard`). Two values, two different computations, two different moments — this structurally explains the reported "0:04 vs 0:05." Making the header defer to the server-persisted value requires `confirmEndRun` (App.tsx ~1173-1235) to update `dnfResult` state after `refreshLeaderboard` resolves, which risks a visible number-flip a beat after the DNF screen renders if not handled carefully (e.g., skeleton the header time until the server row lands, don't flip a rendered number). Add App.tsx to the files list and call out this UX-jank risk explicitly as an implementation constraint.

5. UNRESOLVED PRODUCT QUESTION BEFORE THIS IS IMPLEMENTABLE: what does Results show when the just-finished run is a *worse* repeat (not the account's personal best)? `boardSnippet.ts`'s own doc comment for `boardSnippetRowsFromLeaderboard` states current behavior is deliberate: "'you' means the exact run that just ended (highlightRunId)... a repeat attempt should highlight only itself, matching the pre-refactor behavior." If change (1) collapses Results' snippet to best-rank-per-account, a just-finished non-personal-best run has no matching row after dedup — `BoardSnippet.tsx`'s `yourRow = rows.find(isYou)` returns null and the results screen silently shows no "you" row at all, directly violating PKG-03's own acceptance criterion #4 ("the just-finished run shows... its own board row"). This must be resolved as a product decision (e.g., Results always shows the literal just-raced run's true rank/time even when it isn't the account's placement, with a "not your best — your best is #X" note) before this is buildable as described; flag it back to design/brainstorming, don't let the builder invent it ad hoc.

6. ADD EXPLICIT TEST REQUIREMENTS (none are named in the package): (a) a parity/contract test between whatever dedup mechanism ships and `listChallengePlacements`'s existing tie-break, using an account with a better-later run (the package's own risk note names this case but doesn't require a test for it); (b) `pathsUnlocked` gating tests — never-played (hidden even though other players' paths exist), DNF-only/abandoned-only (still hidden, per invariant 5's "finished, not merely started/DNF'd"), completed (visible); (c) a regression test for finding 5 above (just-finished non-personal-best run must still render on Results); (d) a test that Challenge Detail's new "(you)" row matches `BoardSnippet`'s existing `.is-you` styling/DOM shape (it's copying an established pattern, should be pinned).

7. LOW-RISK, KEEP AS-IS: deleting the unconditional "SERVER TRACKED"/"REPEAT RUN" badges (LeaderboardList.tsx:36-47, confirmed always-rendered with no gating) and adding the `.is-you`/"(you)" treatment to LeaderboardList (confirmed absent today — no isYou/highlight logic exists in the component at all) are both simple, self-contained, low-risk, and should proceed as designed without amendment.

REASONING: Read the spec (docs/superpowers/specs/2026-07-18-ux-redesign-modes-design.md) and all five files in the package plus the server query layer (src/server/d1TrackingRepository.ts) and App.tsx's data flow to verify implementability.

Confirmed accurate/legitimate: LeaderboardList.tsx unconditionally renders "SERVER TRACKED"/"Historical"/"Repeat run" badges (lines 36-47) and "View winning path" (lines 50-63) with zero gating, ChallengeDetail has no "(you)" highlighting anywhere, and the underlying server query `listLeaderboard` (d1TrackingRepository.ts ~2237-2300) truly has no account-level dedup — every eligible attempt gets its own rank row, exactly as the spec's invariant 2 language admits ("the current listLeaderboard query ranks every eligible row per account with no dedup"). So PKG-03's diagnosis is right and its badge-deletion / is-you-highlight changes (3 and 4) are safe, low-risk, ship-as-designed.

But the core fix — change (1)'s client-side account-level dedup — re-implements logic that ALREADY EXISTS, is ALREADY TESTED, and is ALREADY THE SPEC'S OWN PRESCRIBED ARCHITECTURE (Increment 0: "server-side correctness... fixed placement dedup... before touching App.tsx"). The codebase has a second, already-deduped server query (`listChallengePlacements`/`ChallengeBoardPlacement`) that Boards.tsx and Home already consume, whose own doc comment confirms it's "already invariant-2-correct." Re-deriving the same GROUP BY in TypeScript, in a second place, is exactly the "two sources of truth" risk the package's own risk note flags — except it's not hypothetical, the tie-break logic is genuinely intricate (asymmetric completed-vs-DNF ordering) and re-implementing it client-side is unnecessary when a server fix is both safer and matches the spec's own stated architecture.

Layered on top, I found a real structural gap the package doesn't surface: the existing deduped query has no runId (by design — Boards deliberately never discloses per-run paths), but Challenge Detail needs per-row path disclosure after playing. Neither existing server query alone satisfies both dedup and path-linkage, meaning the correct fix is genuinely a small server change (files not listed in PKG-03), not a client workaround.

I also traced the time-mismatch claim (finding 5) through the actual code and found it's structurally real — client-timer-snapshot-before-server-confirm vs. server-computed abandoned_at elapsed — which validates the reported "0:04 vs 0:05" screenshot evidence, but the fix touches App.tsx's confirmEndRun/dnfResult state (unlisted file), with a real jank risk if the header value flips visibly after mount.

Most importantly, I found an unresolved interaction the package doesn't address: dedicating Results' board snippet to best-rank-per-account, per change (1), can make the literal just-finished run vanish from its own results screen when that run isn't the account's personal best — directly undermining acceptance criterion #4. `boardSnippet.ts`'s own comments confirm the current per-run highlight behavior is deliberate, not accidental, so this collision needs a product decision before a builder can implement it as written.

None of this touches race timing math, the run lifecycle FSM, or App.tsx's routing latches (recovery/back-gesture/share-link landing) — those are untouched and the changes are well-contained to presentation/domain-transform files plus (with amendments) a small server query extension and App.tsx data-wiring. The underlying problems (duplicate accounts on boards, unconditional spoiler paths, debug badges, no self-highlight, DNF double-times) are all real and confirmed in code, so this is amend, not drop — but effort should move from M to M/L, the files list needs App.tsx + services/server files added, and two open questions (path+dedup server shape, non-personal-best repeat-run display) need resolving before a builder starts.
