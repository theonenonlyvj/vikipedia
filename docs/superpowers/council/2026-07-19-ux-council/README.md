# 2026-07-19 UX Council — v0 redesign review

12 expert lenses (ux-flow, visual-hierarchy, art-direction, design-system, gameplay, data-viz, mobile-ergonomics, desktop-adaptation, accessibility, copy-voice, spec-fidelity, first-time-user, competitive-benchmark) reviewed 39 live prod screenshots + the ratified mockups + code; 121 raw findings merged into 13 fix packages (PKG-13 added from the same-night daily-drop incident), each verified by 2 independent judges. Briefs in this directory carry the judge amendments and the binding owner-proxy rulings.

## Systemic themes (council synthesis)

v0 drifted for five compounding reasons, all systemic rather than sloppy. First, there is no shared component or class for the app's most repeated ideas — the same 'go race' action, leaderboard row, stat tile, and card frame were each hand-built per screen (three leaderboard renderers, five race-verb labels, a race CTA that's coral on two screens and unstyled default teal on a third), so every convention decayed the moment a second screen needed it. Second, deliberate incremental scoping shipped as user-visible contradiction: the code literally documents that Boards' 'Today' keeps the pre-redesign challenge selection while Home got the new honest one ('deliberately leaves non-Home consumers untouched'), which is why two nav tabs name two different challenges as today's daily. Third, the ratified mockups were treated as inspiration, not contract — nobody ran a shipped-vs-mockup acceptance pass, so whole structural elements (Home's always-on yesterday board, the live HUD timer row, daily numbering, the countdown, the pinned daily in Browse, dark preview card) silently went missing, and that gap is most of what you're feeling on the landing page. Fourth, the stylesheet grew by copy-paste rather than tokens: ~35 raw-hex bypasses of the declared palette, five hand-rolled clip-path sizes, and blanket selectors whose specificity accidents produced real bugs (the coral-washed Browse cards, the top-pinned How-to-play modal inheriting an iOS fix meant for the password dialog). Fifth, 'desktop is a scale-up' was a spec assumption with nothing to scale from — every mockup is a phone frame — so 1440px is just the phone column recentered in black. The fix packages therefore lean cross-cutting: single sources of truth (one daily selection, one board renderer, one button vocabulary) plus one screenshot-vs-mockup acceptance gate per increment, which is the cheap process change that prevents the next v0 from drifting the same way.

## Deliberately dropped

- **spec-fidelity: Browse state chips (NEW/✓/DNF) never appear for anonymous/guest sessions (P1) + gameplay: Browse challenge cards show no per-account outcome chip (P3)** — Both require the bulk per-account outcome server endpoint already scoped as spec Increment 5 — server work, not a UI-consistency fix, and not shippable inside this cycle. Schedule with the Increment 5 server milestone; PKG-01's pinned daily row restores the most important Browse state signal in the meantime.
- **gameplay: Boards' trend views (7d/30d) are flat text lists with no visual rank/celebration treatment (P2)** — The mockup's card/arrow/celebration treatment only pays off once 30d/lifetime boards have real ranked population (currently 0-1 rows). The actual defects on those rows — the overlap smear, tap target, copy — are fixed by PKG-10 and PKG-11. Revisit the celebration pass next cycle when there's data to celebrate.
- **visual-hierarchy: "Create Challenge" and "Create a random new one" have inverted visual weight (P2) + mobile-ergonomics: Manual 'Create Challenge' controls sit below an unbounded, ever-growing challenge list (P2)** — Both point at the same underlying problem — Browse's create flow was never designed as a flow (placement, validation, emphasis). Patching each symptom separately risks churn; defer as one deliberate 'create flow' design unit next cycle. Neither blocks the owner's consistency verdict.
- **mobile-ergonomics: Race mode's stacked sticky header eats ~1/5 of the phone viewport, and End Run sits in the hardest one-handed reach zone (P3)** — The shipped layout matches the ratified mockup (mockup-race-flow-v3 places End Run top-right), so this is a design-change proposal, not a drift bug. Needs an owner decision before anyone touches it; parking for the next design council.
- **design-system: The "boxed stat" pattern is implemented four different ways with no shared class (P2)** — Pure consolidation refactor; its only screenshot-visible symptom (You's broken desktop stat grid) is fixed in PKG-09, and PKG-04 already touches the shared card chrome. Do the .metric-tile extraction in a housekeeping pass after this cycle so it doesn't collide with PKG-04/PKG-09 CSS edits.
- **design-system housekeeping trio: dead .entry-gate CSS / orphaned classNames (P3), duplicate .sync-retry-panel rule (P3), clip-path tokenization (P3)** — Zero user-visible effect — none of it moves the owner's verdict. Batch as a single CSS-housekeeping commit after PKG-04 lands (doing it before or during would create merge churn in the exact same rules PKG-04 edits).
